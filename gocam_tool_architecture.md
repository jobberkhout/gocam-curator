# GO-CAM Curator CLI — Technical Architecture

## Overview

A Python CLI tool that assists a GO-CAM curator through Phases 2, 4, and 5 of the GO-CAM Curation Pipeline. The tool handles text extraction, GO/ECO term mapping, verification against live databases, and narrative generation for expert validation.

### `gocam extract <file>`

**Phase 2 — Entity & relation extraction from any input.**

Accepts multiple input types:
- **Text files** (`.txt`, `.md`): pasted abstracts, results sections, meeting notes
- **Images** (`.png`, `.jpg`): cartoons, diagrams, whiteboard photos, review figures
- **PowerPoint** (`.pptx`): expert lecture slides — extracts all slides as images + speaker notes as text, analyzes each
- **PDF** (`.pdf`): papers — extracts text and embedded figures

For **images and slides**, a strict visual analysis prompt is used that extracts:
- All proteins/genes visible in the diagram (names, labels, abbreviations)
- All arrows/connections and what they represent (activation, inhibition, transport, etc.)
- Cellular compartments shown (pre/post-synaptic, membrane, cytosol, etc.)
- Any text annotations, figure legends, or labels
- What is NOT shown (gaps: "diagram shows A→B→C but no mechanism for the B→C step")

The tool generates a separate extraction file per input, so you can review AI analysis per source and compare it to your own reading.

For each input type, it extracts:
- Proteins/genes mentioned
- Their described activities and interactions
- Exact quotes as evidence anchors
- Gaps flagged (e.g., "activity not described, only binding mentioned")

**Input:** `processes/<name>/input/paper_bhatt2020.txt`
**Output:** `processes/<name>/extractions/paper_bhatt2020.json`

```json
{
  "source": "paper_bhatt2020.txt",
  "timestamp": "2026-03-04T14:30:00",
  "entities": [
    {
      "name": "CaMKII",
      "gene_symbol": "CAMK2A",
      "mentioned_activities": ["phosphorylates GluA1 at Ser831"],
      "context": "postsynaptic density, hippocampal neurons"
    }
  ],
  "interactions": [
    {
      "source_entity": "CaMKII",
      "target_entity": "GluA1",
      "described_action": "phosphorylates at Ser831",
      "quote": "CaMKII phosphorylated GluA1 at Ser831, increasing single-channel conductance",
      "figure": "Fig. 3B",
      "assay_described": "in vitro kinase assay",
      "causal_type": "DIRECT",
      "confidence_note": "Direct biochemical evidence with specific substrate site"
    }
  ],
  "gaps": [
    "No description of how CaMKII is activated (upstream regulation missing)",
    "TARP interaction mentioned but no functional activity described (binding only)"
  ],
  "questions_for_expert": [
    "Is CaMKII activation in this context Ca2+-dependent or constitutive?",
    "What is TARP's actual molecular function beyond scaffolding?"
  ]
}
```

**Prompt used:** `prompts/extract_text.md` (focused on extraction, no GO-term mapping yet)

**For PowerPoint slides:**

The `pptx_reader.py` service:
1. Extracts each slide as a PNG image
2. Extracts speaker notes as text
3. Sends each relevant slide to Claude Vision with `extract_slides.md` prompt
4. Combines all slide extractions into a summary

```
$ gocam extract expert_lecture.pptx

Processing expert_lecture.pptx...
  → Extracted 24 slides
  → Slide 3: "Presynaptic vesicle cycle" — 8 proteins found
  → Slide 4: "Docking and priming machinery" — 6 proteins found
  → Slide 7: "Endocytosis pathway" — 5 proteins found
  → Slides 1,2,5,6,8-24: skipped (title slides, methods, acknowledgments)
  → Speaker notes extracted: 3 slides with relevant notes

Saved: extractions/expert_lecture_slide03.json
Saved: extractions/expert_lecture_slide04.json
Saved: extractions/expert_lecture_slide07.json
Saved: extractions/expert_lecture_notes.json
Saved: extractions/expert_lecture_summary.json
```

The summary file merges all unique entities and connections across slides, deduplicates, and flags inconsistencies (e.g., "Slide 3 shows protein X activating Y, but Slide 7 shows X inhibiting Y").

### `gocam report`

**Phase 2b — Synthesize all extractions into one reviewable AI rapport.**

After you've run `gocam extract` on multiple inputs (papers, slides, cartoons, meeting notes), this command merges everything into a single structured report. You read this alongside your own notes to catch what you missed — or what the AI got wrong.

**Input:** All files in `processes/<n>/extractions/`
**Output:** `processes/<n>/extractions/REPORT.md`

```markdown

| # | File | Type | Entities found | Interactions |

| 2 | paper_lee2018.txt | text | 5 | 3 |
| 3 | expert_slides_slide03.png | slide | 8 | 6 |
| 4 | expert_slides_slide07.png | slide | 5 | 4 |
| 5 | cartoon_endocytosis.png | image | 7 | 5 |

| Dynamin | DNM1 | 2 | GTPase, vesicle scission | HIGH — enzymatic activity known |

2. AP2 --[recruits]--> Clathrin (Sources: 2, 4, 5 | mechanism unclear)

6. NSF --[ATP hydrolysis]--> SNARE disassembly (Sources: 3 | DIRECT)

### Cross-Source Conflicts

  but paper_bhatt2020 describes PICK1 as promoting internalization.

1. **Upstream trigger:** No source describes what initiates the internalization cascade.
2. **PICK1 function:** All sources describe binding only. No actual MF identified.
3. **GRIP1 function:** Described as scaffold in 2 sources. No enzymatic or regulatory activity.

4. "Slides show PICK1 inhibiting internalization, but Bhatt 2020 suggests it promotes it.

```

This report is your **spiekbriefje voor Fase 3** — but generated from all your sources combined. You review it, cross out what's wrong, add your own insights, and go into the expert meeting prepared.

### `gocam verify`

**Phase 4b — Verify all suggested GO/ECO terms against live databases.**

No AI involved — pure API calls.

For each evidence record:
1. **GO terms:** Query QuickGO API → does the term exist? Is it obsolete? Does the label match? Does the aspect (MF/BP/CC) match what was assigned?
2. **UniProt:** Query UniProt → does the protein exist for the given species? What existing GO annotations does it have?
3. **ECO codes:** Query OLS/EBI → does the ECO code exist? Is the label correct?

**Input:** `processes/<name>/evidence_records/records.json`
**Output:** `processes/<name>/verification/report.json`

## Prompt Architecture

Prompts are stored as separate `.md` files so you can iterate on them without touching code.

**`prompts/system.md`** — Shared across all commands. Contains:
- Your role definition (strict GO curator)
- The NO BINDING rule
- The substrate independence rule
- The car test for perturbation data
- Output format expectations (JSON)

**`prompts/extract_text.md`** — Text extraction. Focus:
- Entity extraction only (no GO mapping yet)
- Quote anchoring with exact sentences
- Gap identification
- Question generation for expert meeting

**`prompts/extract_visual.md`** — Image/cartoon analysis. Focus:
- Read every label, abbreviation, and annotation in the image
- Map every arrow/line to a relationship type (activation, inhibition, transport, structural)
- Identify compartments and spatial organization
- Flag what is NOT shown (missing proteins, missing mechanisms)
- Distinguish between "this is drawn" and "this is proven" — a cartoon is a hypothesis, not evidence
- Do NOT infer molecular functions from arrows alone

**`prompts/extract_slides.md`** — Slide-specific instructions layered on top of `extract_visual.md`:
- Skip title slides, methods, acknowledgments
- Extract speaker notes separately — these often contain details not on the slide itself
- Flag slide-to-slide inconsistencies

**`prompts/report.md`** — Cross-source synthesis. Focus:
- Merge entities across all extraction files, deduplicate by gene symbol
- Count how many sources support each entity/interaction (source triangulation)
- Flag conflicts between sources explicitly
- Generate the meeting questions from identified gaps
- Output as readable Markdown — this is the curator's spiekbriefje

**`prompts/translate.md`** — Phase 4 specific. Your current prompt lives here. Focus:
- Biology → GO term mapping
- Method → ECO code mapping
- All GO IDs marked UNVERIFIED
- De novo term suggestions when needed

**`prompts/narrative.md`** — Phase 5 specific. Focus:
- Evidence records → plain English claims
- Confidence communication
- Flagging uncertainties
- Expert-friendly formatting

Each command loads `system.md` + its specific prompt, injects the user's data, and calls the API.

## API Integration Details

### QuickGO (GO term verification)
```
GET https://www.ebi.ac.uk/QuickGO/services/ontology/go/terms/GO:0004674
→ Returns: label, aspect, isObsolete, definition, synonyms
```

### UniProt (protein lookup)
```
GET https://rest.uniprot.org/uniprotkb/search?query=gene:CAMK2A+organism_id:10090&fields=accession,gene_names,go_id
→ Returns: UniProt ID, existing GO annotations
```

### OLS/EBI (ECO verification)
```
GET https://www.ebi.ac.uk/ols4/api/ontologies/eco/terms?iri=http://purl.obolibrary.org/obo/ECO_0000006
→ Returns: label, definition, isObsolete
```

## Build Order (suggested for Claude Code)

1. **Data models first** (`models/`) — define the Pydantic schemas

3. **File processors** (`pptx_reader.py`, `pdf_reader.py`) — extract slides/images/text from files

8. **`gocam narrative`** — connect to narrative prompt
9. **`gocam status`** — read all process folders and display summary
10. **Polish:** error handling, retries, progress bars

## Key Design Decisions

- **Prompts as files, not code:** You can edit `prompts/translate.md` without touching Python. This is crucial for iterating on the GO-CAM rules.
- **All GO IDs unverified by default:** The LLM suggests, `gocam verify` confirms. Never trust an LLM-generated GO ID without checking.
- **JSON as intermediate format:** Everything between steps is JSON. This makes it parseable, diffable, and future-proof for text analysis.
- **One folder per process:** Your file system IS your database. No external DB needed. Git-friendly.
- **Separation of extraction and translation:** Phase 2 (what does the text say?) is deliberately separate from Phase 4 (what GO terms fit?). This prevents the LLM from hallucinating GO terms during extraction.
- **Visual analysis is extraction, not interpretation:** The `extract_visual.md` prompt extracts what is visible in a cartoon (labels, arrows, compartments) but does NOT assign molecular functions from arrows. A drawn arrow from A→B means "the expert thinks A does something to B" — not "A directly positively regulates B." The interpretation happens in Phase 3 (with the expert) and Phase 4 (translation).
- **The report is your second opinion, not your authority:** `gocam report` gives you a structured AI analysis that you lay next to your own reading. Where they agree, you have confirmation. Where they disagree, you have a question for the expert meeting. The tool is never the final answer — you are.
